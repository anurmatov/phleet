## fleet — multi-agent system

autonomous agent platform built in .NET 10. agents run as docker containers, coordinated via RabbitMQ messaging + Temporal workflows. each agent runs a persistent AI process (Claude CLI or Codex SDK bridge) with streaming NDJSON I/O. provider is configurable per agent (default: claude).

repo: https://github.com/anurmatov/phleet

## architecture

```
src/
├── Fleet.Agent/       — core agent process (telegram bot + multi-provider executor: Claude/Codex)
├── Fleet.Orchestrator/ — agent registry, heartbeat tracking, Temporal workflow polling, MCP + REST + WebSocket (port 3600)
├── Fleet.Temporal/    — workflow orchestration bridge (MCP server, port 3001)
│   ├── Engine/        — universal workflow engine (DB-driven JSON step interpreter)
├── Fleet.Bridge/      — RabbitMQ relay for inter-agent messaging (port 3300)
├── Fleet.Memory/      — semantic memory MCP server (qdrant + configurable embedding: ONNX default or ollama, port 3100)
├── Fleet.Shared/      — shared utilities (RabbitMqConnectionHelper)
└── fleet-dashboard/   — React SPA for agent monitoring and lifecycle management (create/copy/delete agents, full config editing, URL query param pre-fill for new agents)
```

## agent lifecycle

1. orchestrator generates config and instruction files from DB into workspace `.generated/` dir (appsettings.json, .mcp.json, settings.json, roles/{Role}/system.md)
2. container starts → entrypoint.sh runs (provider-conditional auth seeding, git config, workspace setup). for codex providers: also generates `~/.codex/config.toml` from `.mcp.json` with per-server `enabled_tools` whitelist (derived from AllowedTools in appsettings)
3. auth seeded from host mount based on provider — claude: `$FLEET_BASE_DIR/.claude-credentials.json` → `~/.claude/.credentials.json`; codex: `$FLEET_BASE_DIR/.codex-credentials.json` → `~/.codex/auth.json`
4. github app token generated at startup
5. `dotnet Fleet.Agent.dll` starts (loads appsettings.json from `.generated/`, bind-mounted `:ro`)
6. persistent AI process launched based on provider — claude: `claude -p` with optional CLI flags `--effort`, `--json-schema`, `--agents` (loads `.mcp.json` and `settings.json` from `.generated/`); codex: Node.js codex-bridge.mjs using @openai/codex-sdk `runStreamed()` API, system prompt written to `/workspace/AGENTS.md` (Codex SDK project instructions discovery), MCP servers configured via `~/.codex/config.toml` (generated by entrypoint.sh), sandbox mode configurable per agent via CodexSandboxMode (default: danger-full-access)
7. role loaded from `.generated/roles/{Role}/system.md` (generated by orchestrator from DB instructions table)
8. project context loaded from `/app/projects/{Project}/`
9. listens to telegram for commands, executes via AI process

## telegram commands

built-in commands handled by CommandDispatcher (not routed to the AI process):
- `/status` — show agent status, running tasks, queued messages, and active background subagent tasks
- `/cancel` — cancel all running tasks
- `/cancel_bg` — list active background subagent tasks; `/cancel_bg <taskId>` cancels a specific one
- `/run <command>` — execute a shell command directly

## MCP tool permissions (critical — update ALL when changing tools)

tool access is controlled in multiple locations. missing one causes tool-not-found or unauthorized access.

1. **orchestrator DB** — `agent_tools` and `agent_mcp_endpoints` tables define per-agent tool access and MCP server connections. managed via `update_agent_config` MCP tool. after changes, run `reprovision_agent` to regenerate config files (`.generated/appsettings.json`, `.generated/.mcp.json`, `.generated/settings.json`) and restart the container.

2. **workflow C# code** — compiled fleet namespace workflows reference tool names. other workflows are UWE DB-driven (no C# files). compiled fleet workflows:
   - src/Fleet.Temporal/Workflows/Fleet/AuthTokenRefreshWorkflow.cs
   - src/Fleet.Temporal/Workflows/Fleet/ConsensusReviewWorkflow.cs

3. **instructions table in orchestrator DB** — role instructions describe available tools in prose (managed via `update_instruction` MCP tool, served to agents as `.generated/roles/{Role}/system.md`)

4. **MCP server source** — where tools are actually defined (`@mcp.tool` in python, `[McpTool]` in C#)

5. **CEO-only signal blocklist** — `temporal_signal_workflow` MCP tool blocks CEO-gate signals (`merge-approval`, `doc-review`, `design-approval`, `advisory-review`) server-side. these can only be sent via the fleet dashboard (orchestrator REST API `POST /api/workflows/{ns}/signal/{id}`, auth-gated). `human-review` and `escalation-decision` remain agent-sendable.

deploy order when removing tools: fleet first (stop referencing) → MCP server second (remove tool)

`fleet-playwright` runs on `fleet-net`. for browser access to services on the same Docker network, use Docker service URLs (e.g. `http://fleet-dashboard`, `http://temporal-ui:8080`) — public domains behind auth proxies may block automated access.

## RabbitMQ messaging

- exchange: `fleet.tasks` (topic) — agent task delegation
- exchange: `fleet.relay` (fanout) — group chat relay
- exchange: `fleet.orchestrator` (topic) — agent heartbeats + registration
- each agent has a dedicated queue bound to `fleet.tasks`
- auto-recovery enabled with 30s heartbeat, 5s reconnect interval
- `ConnectWithRetryAsync` in Fleet.Shared (exponential backoff 2s→60s)

## DelegateToAgentActivity

core activity for all agent-to-agent delegation via Temporal workflows. publishes a directive to the target agent's RabbitMQ queue, waits for the response, and returns the result.

- auto-retries incomplete responses: when an agent hits context/turn limits mid-task (`isIncomplete=true`), automatically re-delegates with a continuation prompt including the previous partial response. up to 3 retries by default (configurable via `retryOnIncomplete` and `maxIncompleteRetries` params). final result text is the concatenation of all partial responses.
- workflow tag: prepends `[fleet-wf:Type:ID]` to every published directive (initial, re-send, and retry continuations) inside `PublishDirectiveAsync`, so agents can verify the message is a legitimate Temporal delegation.
- memory search hint: auto-appends `Search fleet-memory for '{WorkflowType}' to find any operational context before proceeding.` to every published directive (initial, 5-min re-sends, and retry continuations). `WorkflowType` is read from `ActivityExecutionContext.Current.Info.WorkflowType`. enables automatic discovery of operational context memories via naming convention (memories titled `"{WorkflowType} — operational context"`). gracefully degrades when no matching memory exists.
- cancellation: on activity cancellation, sends targeted HTTP cancel to the target agent via orchestrator REST API (`POST /api/agents/{name}/cancel/{taskId}`). only cancels the specific delegated task, leaving other running/queued tasks unaffected. falls back to RabbitMQ `/cancel` broadcast if orchestrator URL is not configured.
- timeout: fires `TimeoutException` and notifies the CTO agent once after all retries are exhausted (not per-retry).

## temporal workflows

fleet supports multiple Temporal namespaces. each has its own task queue. worker registration is config-driven — each configured namespace gets a uniform worker with all workflows and activities registered. the out-of-the-box stack configures one namespace: `fleet`.

## universal workflow engine (UWE)

UniversalWorkflow — `[Workflow(Dynamic = true)]` handler registered on namespace workers. catches any workflow type name without a statically-compiled handler. loads JSON step tree from orchestrator DB at runtime via `LoadWorkflowDefinition` activity, then interprets it.

- 16 step types: sequence, parallel, loop, branch, break, continue, noop, set_variable, delegate, delegate_with_escalation, wait_for_signal, child_workflow, fire_and_forget, cross_namespace_start, set_attribute, http_request
- template engine: `{{expr}}` interpolation with scopes (input, vars, config) and filters (`| default`, `| extract`, `| json`)
- 2 determinism-safe activities: `LoadWorkflowDefinition`, `LoadWorkflowConfig` (loaded once per execution, replayed from history)
- `WorkflowTypeRegistry` lazy-merges DB-defined types into `temporal_list_workflow_types` with 5-min background refresh
- `break` and `continue` are proper step types used inside branch cases for loop control flow (e.g. `"APPROVED": {"type": "break"}`)
- `noop` is an explicit no-op step — clearer than empty sequences for default branch cases
- branch cases are always `StepDefinition` objects — no dual-mode action pattern
- signal: `escalation-decision` (used by `delegate_with_escalation` steps) with payload `{"Decision":"retry|skip|continue","UpdatedInstruction":"..."}`
- `ResolveArgs` omits null and empty-string values from child workflow args — compiled C# workflows receive null for nullable types instead of crashing on type mismatch
- DB tables: `workflow_definitions`, `workflow_definition_versions`
- REST CRUD: `GET/POST/PUT /api/workflow-definitions`, `POST /api/workflow-definitions/:name/toggle-active`

**compiled fleet namespace workflows:**
- ConsensusReviewWorkflow — multi-agent parallel review. input: `ReviewerAgents` (required), `Synthesizer` (required), `PrUrl`, `Title`, `Body`, `AgentPerspectives`. throws `ArgumentException` if `ReviewerAgents` or `Synthesizer` null/empty
- AuthTokenRefreshWorkflow — scheduled (every 30 min) centralized multi-provider OAuth token refresh. performs direct HTTP refresh via `RefreshAuthTokenActivity` for each configured provider (claude, codex) with per-provider failure isolation (if one fails, others still run). temporal-bridge writes refreshed credentials directly to source files via bind mounts (`$FLEET_BASE_DIR/.claude-credentials.json`, `$FLEET_BASE_DIR/.codex-credentials.json`), then broadcasts via `BroadcastTokenUpdateActivity` (includes `provider` field so agents filter updates by their own provider) to `fleet.relay` fanout (agents only). agents apply tokens via `GroupBehavior.ApplyTokenUpdateAsync` and restart their CLI process. orchestrator is not involved in credential flow. claude uses 45-min refresh threshold; codex uses OAuth endpoint with JWT exp-based expiry detection (1-day refresh threshold). OAuth client IDs are configurable via `AuthTokenRefresh` config section.
  - input: `{"Providers": ["claude", "codex"]}` (optional, defaults to `["claude"]`)

**UWE fleet workflows (seeded in `seed.example.json`):**
- FireAndForgetTaskWorkflow — lightweight fire-and-forget delegation to a single target agent. no result-delivery step. useful for scheduled self-delegation.
- TaskDelegationWorkflow — general-purpose ad-hoc task delegation to any agent. delegates via `DelegateToAgentActivity`, waits for response (configurable timeout), delivers result to the CTO agent. prefer over relay when tracking and guaranteed delivery matter.
- MemoryStoreRequestWorkflow — gates memory writes behind the CTO agent's review via delegation loops. requesting agent submits proposed memory content, CTO reviews via delegation (searches for duplicates, evaluates quality), stores/updates directly via memory MCP tools. supports back-and-forth REVISE loop with requesting agent. agents start this workflow via `temporal_start_workflow` (namespace=fleet, task_queue=fleet).
  - input: `{"Type":"...","Title":"...","Content":"...","Project":"...","Agent":"..."}` (Agent field used for dynamic delegation target on REVISE path)
- UwePrImplementationWorkflow — issue → branch → code → PR → review → merge-gate → merge.
  - phases: 1=implement, 2=consensus review loop, 2.5=unresolved comment check, 3=CEO merge-approval gate, 4=merge, 5=doc maintenance (fire-and-forget)
  - signals: `human-review` `{"Decision":"approved|changes_requested|rejected","Comment":"..."}`, `merge-approval` `{"Decision":"approved|changes_requested|rejected","Comment":"..."}`
  - merge-gate loop: CEO sends `changes_requested` → consensus review evaluates concern → if agreed, dev agent fixes + full review re-runs; if reviewers still approve, CEO notified with reasoning; loop repeats up to a configurable maximum
  - after merge: `UweDocMaintenanceWorkflow` fires as fire-and-forget child (Phase 5)
- UweDesignWorkflow — creates or refines a well-specified GitHub issue with multi-agent consensus review, then gates on CEO design-approval. target agent creates/refines the issue, then `ConsensusReviewWorkflow` runs as a child with configurable reviewers. once consensus approves, goes to CEO gate.
  - input: `Description` (required), `ExistingIssueNumber` (optional — refine instead of create), `Repo` (required), `TargetAgent` (required), `ConsensusAgents` (required), `MaxConsensusRounds` (default: 3)
  - signal: `design-approval` `{"Decision":"approved|changes_requested|rejected","Comment":"..."}`
  - consensus loop: target creates/refines → `ConsensusReviewWorkflow` reviews → if `changes_requested`, feed reasoning back to target and loop (max `MaxConsensusRounds`). on exhaustion → cancel
  - CEO gate (no round limit): on `changes_requested`, feed feedback to target and re-enter consensus loop. hourly reminders, never auto-cancels
- UweDesignToPrWorkflow — thin orchestrator: `UweDesignWorkflow` child → `UwePrImplementationWorkflow` abandoned child. use for tasks that need a well-specified issue before implementation begins.
  - input: `Description` (required), `ExistingIssueNumber` (optional), `Repo` (required), `DesignAgent` (required), `ImplementAgent` (required), `ConsensusAgents` (required), `MaxConsensusRounds` (default: 3)
  - phases: 1=`UweDesignWorkflow` child (consensus review + CEO design-approval gate), 2=fire `UwePrImplementationWorkflow` as abandoned child with approved issue number
- UweDocMaintenanceWorkflow — post-merge doc and memory update (PR-based: developer creates PRs per repo, CEO doc-review gate: approve=merge all PRs, reject=close all PRs, `changes_requested`=push fixups to existing branches; memory updates direct via MCP)

## temporal search attributes

workflows set custom search attributes for dashboard visibility. these must be registered once per namespace on the temporal server (`temporal operator search-attribute create`).

| attribute | type | set by | purpose |
|-----------|------|--------|---------|
| IssueNumber | Int | UwePrImplementationWorkflow, UweDesignWorkflow, UweDesignToPrWorkflow | GitHub issue number (from input or created) |
| PrNumber | Int | UwePrImplementationWorkflow, UweDocMaintenanceWorkflow | PR number (parsed from agent output or passed from parent) |
| Repo | Keyword | UwePrImplementationWorkflow, UweDocMaintenanceWorkflow, UweDesignWorkflow | repo in org/name format |
| DocPrs | Keyword | UweDocMaintenanceWorkflow | space-separated `org/repo#N` refs for doc PRs created |
| Phase | Keyword | all workflows with signal gates | current signal wait phase (e.g. `human-review`, `merge-approval`, `design-approval`). dashboard filters signal buttons to only show the one relevant to the current phase |

the orchestrator's `TemporalPollerService` reads these attributes when polling and includes them in `WorkflowSummary`. the dashboard renders clickable issue/PR links on workflow cards and filters signal buttons by Phase. all upserts are wrapped in `try/catch` — failures are non-fatal and never crash workflows. `SearchAttributeInitializer` auto-registers all attributes on bridge startup.

## config hierarchy (priority order)

1. environment variables (from `.env`)
2. `.generated/appsettings.json` (generated by orchestrator from DB, bind-mounted `:ro` at provision time). includes Agent section with Provider, Effort, JsonSchema, AgentsJson, CodexSandboxMode fields that map to CLI flags in `ClaudeExecutor` or env vars in `CodexExecutor`
3. `src/Fleet.Agent/appsettings.json` (baked fallback)

## orchestrator REST + WebSocket API

auth: mutating endpoints (POST/PUT/DELETE) require `Authorization: Bearer <token>`. GETs, WebSocket (`/ws`), MCP (`/mcp`), and `/health` are exempt. if `Orchestrator:AuthToken` is empty, auth is disabled (backward-compat mode). token is provisioned via `ORCHESTRATOR_AUTH_TOKEN` environment variable.

REST endpoints:
- `GET /api/agents` — all agent states (merges DB records with in-memory heartbeat registry; DB-only agents appear with status `not_provisioned`)
- `GET /api/agents/{name}` — single agent state
- `POST /api/agents` — create a new agent (insert into DB; `provision=false` skips container start, agent appears as `not_provisioned`)
- `DELETE /api/agents/{name}` — delete agent (deprovision container + remove DB records)
- `GET /api/agents/{name}/history` — task history (last 100 completed tasks)
- `GET /api/agents/{name}/config` — full DB config (model, memory, tools, projects, MCP endpoints, instruction assignments)
- `PUT /api/agents/{name}/config` — update agent DB config (all scalar fields + replace-all for tools, projects, MCP endpoints, networks, env refs, telegram users/groups, instructions)
- `POST /api/agents/{name}/restart` — restart agent container
- `POST /api/agents/{name}/stop` — stop agent container
- `POST /api/agents/{name}/start` — start agent container
- `POST /api/agents/{name}/cancel` — cancel all running tasks on an agent (proxies via container name on Docker network, falls back to host port if no container name)
- `POST /api/agents/{name}/cancel/{taskId}` — cancel a specific foreground task by bridge taskId. used by `DelegateToAgentActivity` for targeted cancellation
- `POST /api/agents/{name}/cancel_bg/{taskId}` — cancel a specific background subagent task
- `POST /api/agents/{name}/reprovision` — deprovision + provision from DB config (optional `?image=` query param for image override)
- `POST /api/agents/{name}/restart-with-version` — restart agent with specific instruction version (body: `instructionName`, `versionNumber`, `pin`)
- `POST /api/agents/reprovision-running` — bulk reprovision all currently running agents (optional `?image=` query param)
- `GET /api/workflows` — active Temporal workflows across all namespaces
- `GET /api/workflow-types` — merged workflow type registry: hardcoded C# types (with JSON input schemas) + active UWE DB types. used by dashboard `StartWorkflowModal`
- `POST /api/workflows/start` — start a new Temporal workflow (body: `workflowType`, `namespace`, `taskQueue`, `workflowId?`, `input?`). auto-generates workflowId as `{type}-{timestamp}` if omitted. auth required
- `POST /api/workflows/{ns}/{id}/cancel` — cancel a running workflow (graceful)
- `POST /api/workflows/{ns}/restart/{id}` — restart a workflow (reads original type + input from history, starts fresh execution; `?terminateExisting=true` to kill old run first)
- `POST /api/workflows/{ns}/{id}/terminate` — terminate a running workflow (force kill)
- `POST /api/workflows/{ns}/signal/{id}` — send a signal to a running workflow
- `GET /api/workflows/{ns}/history/{id}` — workflow execution history (filtered activity events with decoded payloads, 300 event cap)
- `GET /api/workflows/failures` — recently failed/terminated/canceled workflows (last 1 hour)
- `GET /api/workflows/completed?hours=24` — recently closed workflows across all namespaces (Completed, Failed, Canceled, Terminated) with search attributes and CloseTime. Default 24h window, max 72h, capped at 50 results
- `GET /api/signals` — signal registry (workflow type → available signals with payloads)
- `GET /api/search-attributes` — registered search attribute names (from `SearchAttributeTypeRegistry` in Fleet.Shared)
- `GET /api/namespaces` — list of active Temporal namespaces (from connected clients or config fallback). used by dashboard to dynamically populate namespace dropdowns
- `GET /api/schedules` — list all Temporal schedules across all configured namespaces (cheap fields: `scheduleId`, `namespace`, `workflowType`, `cronExpression`, `paused`, `memo`)
- `GET /api/schedules/{ns}/{id}` — describe a single schedule (full detail: next/last run times, last workflow ID, input)
- `POST /api/schedules` — create a new schedule (body: `scheduleId?`, `namespace`, `workflowType`, `taskQueue?`, `cronExpression`, `input?`, `memo?`, `paused?`). auto-generates scheduleId as `{workflowType}-{8-char-guid}` if omitted

- `POST /api/schedules/{ns}/unpause/{id}` — unpause a schedule
- `POST /api/schedules/{ns}/trigger/{id}` — trigger an immediate run
- `DELETE /api/schedules/{ns}/{id}` — delete a schedule
- `GET /api/workflow-definitions` — list workflow definitions (active only by default; `?includeInactive=true` returns all with `isActive` field)
- `GET /api/workflow-definitions/{name}` — get single definition with full JSON step tree + version history (works for inactive definitions)
- `POST /api/workflow-definitions` — create workflow definition
- `PUT /api/workflow-definitions/{name}` — update definition (auto-saves previous version; nullable DTO — omitted fields are preserved)
- `POST /api/workflow-definitions/{name}/toggle-active` — enable/disable a workflow definition (body: `{ isActive: bool }`)
- `GET /api/instructions` — list all instructions with current version, `isActive` flag, and agent assignments
- `GET /api/instructions/{name}` — full instruction content with all versions
- `POST /api/instructions` — create a new instruction with initial v1 content (body: `name`, `content`, `reason?`, `createdBy?`)

## orchestrator MCP tools

container lifecycle management (CTO agent only):
- `create_agent(agent_name, role, model, display_name?, memory_limit_mb?, container_name?, provider?)` — creates a DB record only, no container. use `manage_agent_*` tools to configure, then `provision_agent` to start.
- `provision_agent(agent_name)` — creates and starts a container from DB config (image, memory, env, binds, networks). fails if container already exists.
- `deprovision_agent(agent_name)` — stops and removes the container. workspace volume data preserved.
- `reprovision_agent(agent_name)` — deprovision + provision in sequence. use after config changes (model, tools, memory).
- `restart_agent_with_version(agent_name, instruction_name, version_number, pin=false)` — reprovision with a specific instruction version. `pin=false` is temporary (next normal reprovision reverts to current DB version). `pin=true` updates `CurrentVersion` in DB making it sticky.

agent management:
- `list_agents`, `get_agent_status`, `get_agent_history` — read agent state
- `restart_agent`, `stop_agent`, `start_agent` — container lifecycle (simple restart, no reprovisioning)
- `system_health` — fleet-wide health summary
- `agent_logs` — fetch container logs

configuration management:
- `list_agent_configs`, `get_agent_config`, `update_agent_config` — DB-stored agent config (all scalar fields: model, memory, image, provider, codex_sandbox_mode, permission_mode, max_turns, work_dir, proactive/group settings, show_stats, prefix_messages, tts_service_url, effort, json_schema, agents_json, host_port, auto_memory_enabled + replace-all for tools and projects)
- `manage_agent_mcp_endpoints(agent_name, action, mcp_name, url?, transport_type?)` — add/update/remove MCP endpoint connections per agent
- `manage_agent_networks(agent_name, action, network_name)` — add/remove Docker networks per agent
- `manage_agent_env_refs(agent_name, action, env_key_name)` — add/remove environment variable key references per agent
- `manage_agent_telegram_users(agent_name, action, user_id)` — add/remove allowed Telegram user IDs per agent
- `manage_agent_telegram_groups(agent_name, action, group_id)` — add/remove allowed Telegram group IDs per agent
- `manage_agent_instructions(agent_name, action, instruction_name, load_order?)` — add/update/remove instruction assignments per agent
- `list_instruction_versions`, `update_instruction`, `rollback_instruction`, `diff_instruction_versions` — versioned instruction management
- `preview_agent_provision` — diff desired (DB) vs actual (Docker) container spec
- `ensure_networks_exist` — create missing Docker networks from DB

repository management:
- `list_repositories(includeInactive?)` — list managed GitHub repos (active only by default)
- `manage_repository(action, name, full_name?)` — add/update/remove a managed repo. action: `add | update | remove`

## dashboard alerting

the dashboard has a built-in alerting system with detection types:
- agent-dead / agent-stale — detected from WebSocket state transitions
- task-stuck — periodic check every 60s, fires if same task running > 30 minutes
- workflow-failed — polls `/api/workflows/failures` every 60s
- info — informational messages (e.g. "workflow no longer in active list")

alerts appear as toasts (bottom-right, auto-dismiss 8s) and persist in a dismissible alert panel in the header bar. alerts store optional structured metadata (workflowId, workflowNamespace, agentName) for entity linkage. toast and alert-panel items with metadata show a clickable "view" / "view workflow" / "view agent" button that navigates to the relevant view and scroll-to-highlights the entity. all client-side, no Telegram integration.

## dashboard features

- agent cards show background subagent tasks (type badge, description, elapsed time, summary) with per-task cancel button. data comes from `BackgroundTasks` field in heartbeat payload (populated from Claude SDK `task_started`/`task_progress`/`task_notification` NDJSON events).
- global search: `Cmd+K` / `Ctrl+K` opens search in header bar. searches agents (by name, displayName, shortName) and workflows (by type, ID, issue number, PR number). click result → navigate to view + scroll-to-highlight with flash animation.

## orchestrator database

MySQL 8.0 (`fleet-mysql` container) stores agent configuration. EF Core 9 + Pomelo MySQL provider. connection is optional — orchestrator falls back gracefully when no connection string is set.

schema (`fleet_orchestrator` database):
| table | purpose |
|-------|---------|
| agents | core agent registry (name, role, model, memory limit, container name, enabled, Image, Provider, CodexSandboxMode) + behavior fields (PermissionMode, MaxTurns, WorkDir, ProactiveIntervalMinutes, GroupListenMode, GroupDebounceSeconds, ShortName, ShowStats, PrefixMessages, TtsServiceUrl, AutoMemoryEnabled) + claude CLI flags (Effort, JsonSchema, AgentsJson) + networking (HostPort — optional override for orchestrator→agent HTTP proxy; if null, computed deterministically as `8080 + agent.Id`) |
| agent_tools | tools per agent (many-to-many, with enabled flag) |
| agent_projects | projects per agent |
| agent_mcp_endpoints | MCP server connections per agent (name, url, transport type) |
| agent_telegram_users | allowed Telegram user IDs per agent |
| agent_telegram_groups | allowed Telegram group IDs per agent |
| instructions | named instruction blocks (e.g. `base`, `co-cto`) with current version and `IsActive` flag (default true, for archive/unarchive) |
| instruction_versions | version history per instruction (content, created_by, reason). last 20 kept |
| agent_instructions | agent → instruction mapping with `load_order` for composing multiple instructions |
| agent_networks | docker networks per agent (e.g., `fleet-net`) |
| agent_env_refs | secret key name references per agent (no actual values stored) |
| project_contexts | named project context blocks with current version and `IsActive` flag (default true, for archive/unarchive) |
| project_context_versions | version history per project context |
| workflow_definitions | DB-driven workflow definitions (name, namespace, task_queue, JSON step tree, version, is_active) |
| workflow_definition_versions | version history for workflow definitions (rollback support) |
| repositories | managed GitHub repos (name, full_name, is_active) used as picker source in workflow start forms |

## orchestrator configuration

| section | key | purpose |
|---------|-----|---------|
| ConnectionStrings | OrchestratorDb | MySQL connection (empty = DB disabled) |
| RabbitMq | Host, Exchange | heartbeat consumer |
| Temporal | Address | temporal server (empty = poller disabled) |
| Temporal | Namespaces | namespaces to poll (no default — must be explicitly configured via env vars in docker-compose) |
| Temporal | PollIntervalSeconds | poll interval (default: 15) |
| Orchestrator | AuthToken | bearer token for mutating endpoint auth (empty = auth disabled) |
| Provisioning | EnvFilePath | path to `.env` file for secret resolution (default: `$FLEET_BASE_DIR/.env`) |
| Provisioning | BaseDir | fleet base directory for `./` path expansion (required — throws `InvalidOperationException` if not configured. set via `FLEET_BASE_DIR` env var) |
| Provisioning | AgentImage | default Docker image for agent containers (default: `fleet:agent`) |

deploy note: orchestrator runs as a Docker container on `fleet-net`. `Temporal__Address` is set to `temporal-server:7233` for workflow polling. Docker socket is mounted for container provisioning.
